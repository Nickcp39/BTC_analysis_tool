# BTC 价格预测 / 减半→峰值对比 小程序 设计文档

## 设计原则（优先遵守）

- **时间优先于价格**：BTC 预测中 **时间的准确度高于价格**；价格仅用于辅助（如区间参考、回撤倍数等）。多模型融合与产品展示时应以时间锚点/时间窗口为主，价格区间为辅。

---

## 1. 产品目标

- **核心功能**：用户打开程序后，自动抓取最新 BTC 价格数据，按既有 Python 逻辑处理并出图。
- **主图形态**：与现有产出一致——**减半→峰值对齐（含峰后）：波动退火后**，即多年度曲线（如 2017、2021、2025）以「锚点=峰值日」对齐，Y 轴为相对锚点的涨跌百分比，历史周期按波动退火系数缩放后与参考周期（当前/2025）对比。
- **动态演进**：图表随时间推进自动更新：新数据拉取 → 重算 → 重绘；参考周期（如 2025）的曲线会随日期延伸，历史周期保持固定逻辑。
- **发布形态**：支持 **App Store 上架** 或 **网页版** 两种形态（可先做其一，架构上预留另一形态）。

## 2. 现有 Python 逻辑摘要（须保持一致）

以下逻辑来自 `code/step01_halving_to_peak.py` 与 `stepA1_get_data_real_time.py`，新产品需在语义上与之一致。

### 2.1 数据来源与获取

| 用途       | 当前实现                         | 说明 |
|------------|----------------------------------|------|
| 当前周期   | FRED `CBBTCUSD` CSV → 存 Excel  | `stepA1_get_data_real_time.py`：`https://fred.stlouisfed.org/graph/fredgraph.csv?id=CBBTCUSD` |
| 历史 2017 | `data/btc_2015.xlsx` 两列(时间, 价格) | 覆盖 2016 减半到 2017 峰值及之后 |
| 历史 2021 | `data/btc_2021.xlsx` 两列(时间, 价格) | 覆盖 2020 减半到 2021 峰值及之后 |

- 小程序需：**打开时自动拉取最新数据**（至少当前周期等价于 FRED 最新）；历史周期可内置或从同一数据源按日期切片。

### 2.2 时间锚点与窗口

- **减半日**：2016-07-09、2020-05-11、2024-04-20（固定）。
- **峰值日**：
  - 2017：固定 `2017-12-19`。
  - 2021：`None` 时取 2021 年内最高价对应日期。
  - 2025：当前为假设值 `2025-08-15`；随产品演进可改为「至今最高」或可配置。
- **窗口**：从减半日起，到 `峰值日 + POST_DAYS`（如 60），向前最多取 `MAX_PRE_DAYS`（如 300）天。
- **锚点**：X 轴「相对天数」以**峰值日为 0**；Y 轴为相对该日价格的涨跌百分比。

### 2.3 核心计算

1. **日频序列**：原始 (ts, price) → 按日 resample，缺日 ffill。
2. **窗口切片**：`window_halving_to_peak(series, halving, peak, post_days, max_pre)`。
3. **相对锚点曲线**：`pct_curve(series, anchor_date)` → `rel_day`、`dd_pct = (price/anchor_price - 1)*100`。
4. **峰前波动（pre_std）**：峰前若干天（如 90 天）的 `dd_pct` 的 std，用于缩放（可选）。
5. **缩放系数（波动退火）**：
   - `scale_factor(..., method="manual", alpha=0.5)` 时：`(level_new/level_old)**alpha`。
   - 当前常量：VOL_LEVEL_2017=9, VOL_LEVEL_2021=3, VOL_LEVEL_2025=1 → 2017 缩放、2021 缩放相对 2025。
6. **出图**：2025 不缩放（参考）；2017/2021 用上述系数缩放后与 2025 同图，X=0 画竖虚线，图题「减半→峰值对齐（含峰后）：波动退火后」。

逻辑一致性要求：**同一输入数据 + 同一锚点/窗口/退火参数，应得到与现有 Python 相同的数值结果与图表形态**。

## 3. 技术方案概览

### 3.1 方案 A：网页版（推荐先行）

- **优点**：迭代快、无需审核、易分享、可同一套后端服务未来给 App 用。
- **技术选型建议**：
  - 前端：React / Vue / 纯 HTML+JS，用 ECharts 或 Chart.js 等画「相对天数 × 相对锚点%」折线图。
  - 后端：轻量 API（Node / Python FastAPI 等），负责：拉取 FRED（及可选历史数据）→ 按上面 2.2/2.3 逻辑计算 → 返回各年 `rel_day`、`dd_pct`（及缩放后序列）、缩放系数、锚点信息。
  - 逻辑一致：**核心算法放在后端，用 Python 或移植为与 Python 完全一致的公式（如 JS/TS）**，避免前端重写产生偏差。

### 3.2 方案 B：App Store 原生 / 跨平台

- **优点**：可上架 App Store，离线缓存、推送等体验更好。
- **技术选型建议**：
  - 原生：Swift (iOS)；或跨平台：React Native / Flutter。
  - 数据与计算：优先**调用与网页版同一后端 API**，保证逻辑一致；若必须端侧算，则用与 Python 完全一致的公式（或嵌入 Python 引擎/WebAssembly 编译的同一套算法）。
  - 图表：用平台图表库（如 Swift Charts、RN/Flutter 图表组件），坐标与图例需与现有图一致（相对天数、相对锚点%、图题、图例带缩放系数）。

### 3.3 混合策略（推荐）

- 先做 **网页版 + 统一后端 API**，核心计算与 Python 同源（同一仓库内 Python 模块或移植为确定性的 JS/TS）。
- 再基于同一 API 做 **App**（或 PWA），保证「打开即拉数、处理、出图」与「随时间变化」行为一致。

## 4. 架构与数据流

```
[用户打开] 
    → [拉取最新数据]（FRED + 可选历史或内置历史）
    → [解析为日频 (date, price)]
    → [按周期：减半/峰值/窗口 切片]
    → [计算相对锚点曲线 rel_day, dd_pct]
    → [计算缩放系数（波动退火）]
    → [输出：2025 原始 + 2017/2021 缩放后序列]
    → [渲染：减半→峰值对齐（含峰后）图]
```

- **动态更新**：每次打开或刷新时重新执行上述流程；若 2025 峰值改为「至今最高」，则每次用最新数据重算峰值日再对齐。
- **历史数据**：2017/2021 可预置或由后端从同一 FRED 长序列按日期范围切片，保证与现有 Excel 输入等价。

## 5. 逻辑一致性实现要点

- **公式与常数**：将 `scale_factor`、`pct_curve`、`window_halving_to_peak`、`pre_std`、`VOL_LEVEL_*`、`VOL_ALPHA`、`POST_DAYS`、`MAX_PRE_DAYS` 等集中为单一「计算规范」（如文档或共享代码），后端/前端严格按此实现。
- **数值验证**：用现有 Python 的某日输出（如 `halving_to_peak_aligned.csv`、`halving_to_peak_metrics.txt`）做快照测试，新实现同输入应得到相同缩放系数与曲线数值（在浮点容差内）。
- **峰值 2025**：当前为固定日；若改为「滚动至今最高」，需在设计里明确规则（例如：仅用 2024 减半后的数据，取该区间内最高价日），并在文档中写清以便与 Python 日后对齐。

## 6. 数据与配置

- **当前周期数据**：必须「打开即拉取」最新（如 FRED CSV），不得依赖本地 Excel 更新。
- **历史周期**：可内置 2017/2021 的 (date, price) 或由后端从 FRED 长序列生成；若 FRED 覆盖不足，需保留与现有 `btc_2015.xlsx` / `btc_2021.xlsx` 一致的数据源或导出格式。
- **配置项**：峰值日（2017 固定、2021 自动/固定、2025 固定或规则）、POST_DAYS、MAX_PRE_DAYS、波动退火参数等建议可配置（环境变量或后台配置），默认与当前 Python 一致。

## 8. 给定日期 → 价格区间（逆向推导）

- **需求**：用户输入一个目标日期（如 2026 年 4 月），用与「峰值时间窗口 + 峰值/底价格区间」同一套拟合逻辑，**逆向**推出该日期的预计价格区间（如 8~12 万）。
- **正向已有一致结论**：例如「峰值时间窗口 2025-08-15 ~ 2025-10-15」「峰值价格区间 12~16 万」「顶→下一底约 1.2 年」「下一底区间约 4~6 万」。
- **逆向逻辑（时间优先，价格为辅）**：
  1. **时间定位**：用 step02/4 fusion 得到峰值时间中心与窗口、顶→底时长（如 1.2 年）、下一底时间中心。
  2. **阶段判断**：目标日期落在「峰前 / 峰窗内 / 峰后→底 / 底后」哪一段。
  3. **价格区间**：
     - **峰窗内**：直接给峰值价格区间。
     - **峰后→底**：按「距峰已过天数 / 顶→底总天数」得到进度，在「峰值区间 → 下一底区间」之间线性（或按历史回撤形态）插值，得到该日期的价格上下界。
     - **峰前**：按「减半→峰」进度在「上一底→峰值区间」之间插值（可选）。
     - **底后**：给下一底区间或下一周期逻辑（可选）。
- **实现**：在 `code/step03_date_to_price.py` 中实现：输入 (年, 月)，输出 (价格下限, 价格上限, 阶段说明)。
- **原则**：时间由多模型融合给出（准确度优先），价格仅作辅助区间，不做点估为主的产品呈现。
